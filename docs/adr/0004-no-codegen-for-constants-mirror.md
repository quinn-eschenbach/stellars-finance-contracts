# Constants mirror is per-line comments, not codegen

Four numeric constants are mirrored between `shared/src/constants.rs` (on-chain) and `packages/config/constants.ts` (off-chain services): `BPS` / `BPS_DENOMINATOR`, `SECONDS_PER_YEAR`, `SECONDS_PER_LEDGER`, and `PRECISION`. Each TS value carries a `// Mirror of shared::constants::X` comment; drift is caught in code review rather than by a build-time codegen step. The architecturally cleaner alternative was a Rust build script that emits a generated TS file, eliminating drift structurally. We rejected that option because the four mirrored values are mathematical facts (BPS = 10_000, SECONDS_PER_YEAR = 31_536_000), Stellar network parameters (SECONDS_PER_LEDGER), or fundamental on-chain choices (PRECISION = 1e7) whose change would break stored prices irreversibly — so the drift risk approaches zero. The trajectory of the codebase further argues against codegen: on-chain constants grow with protocol bounds (max_leverage, deviation ceilings) while off-chain constants grow with service tunings (oracle poll intervals, keeper fee budgets, SSE buffers), so the shared subset stays small. Revisit if the shared subset grows past ~10 values, or if a constant becomes genuinely tunable (changes more than once per year), at which point comment-based discipline is no longer sufficient and codegen pays for itself.
