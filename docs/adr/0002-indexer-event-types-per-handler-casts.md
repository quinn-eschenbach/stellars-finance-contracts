# Indexer event types are per-handler casts, not codegen

The indexer's seam between `scValToNative` decoding and DB persistence is `ParsedEvent.data: Record<string, any>`. Each handler declares a local `interface FooData` mirroring the contract's `#[contractevent]` struct and casts `event.data as FooData` once at the top of the function — typed within the handler, untyped at the seam. The architecturally cleaner alternative was to codegen typed events from the contract spec (the way function bindings are generated by `stellar contract bindings typescript`), making drift impossible. We rejected that option because the indexer is the only consumer of these events today; a codegen pipeline for a one-adapter system buys safety against drift the codebase isn't currently exposed to. Hand-written types in handlers were rejected for the opposite reason: they take on maintenance cost without a second consumer to amortize against. Revisit when a second consumer emerges (for example, SSE broadcasting typed events end-to-end, a separate analytics service, or a read-side that lives outside this repo) — at that point the codegen investment pays for itself across consumers and the per-handler casts become the brittleness.
